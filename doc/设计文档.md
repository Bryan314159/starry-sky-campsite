# 设计方案：收藏夹可视化浏览器扩展

## Context

基于《项目梗概》中的需求，需要将浏览器书签转化为一个沉浸式 3D 场景的新标签页扩展。用户每次打开新标签页时，看到的是一个营地和星空的世界，通过场景化交互浏览书签。前后端（数据层与表现层）分离设计。

## 2. 架构概览：前后端分离

```
┌─────────────────────────────────────────────────────┐
│                  Browser Extension                   │
│                                                      │
│  ┌──────────────┐       ┌──────────────────────────┐│
│  │  后端 (SW)    │ ────▶ │  前端 (New Tab Page)     ││
│  │              │  msg  │                          ││
│  │ · 书签读取    │       │  · Three.js 3D 渲染      ││
│  │ · 使用统计    │       │  · 场景管理               ││
│  │ · 数据缓存    │       │  · 交互动画               ││
│  └──────────────┘       └──────────────────────────┘│
│         │                          │                │
│         ▼                          ▼                │
│  chrome.bookmarks            chrome.tabs             │
│  chrome.storage              window.location         │
└─────────────────────────────────────────────────────┘
```

**后端职责：**
- 通过 `chrome.bookmarks` API 读取书签树
- 解析为两级结构（文件夹 → 书签项）
- 追踪书签打开次数（频率统计）
- 通过 `chrome.runtime.onMessage` 响应前端的查询请求

**前端职责：**
- Three.js 渲染 3D 场景
- 场景一（营地路牌）和场景二（星空）的构建与交互
- 场景过渡动画（小猫行走）
- 响应用户点击，打开网址或切换场景

**通信方式：**
- 前端通过 `chrome.runtime.sendMessage()` 请求书签数据
- 后端 Service Worker 通过 `chrome.runtime.onMessage` 监听并响应

## 3. 数据模型（shared/types.ts）

```typescript
// 书签分类（一级）
interface BookmarkCategory {
  id: string;           // chrome.bookmarks 的 folder id
  title: string;        // 文件夹名称
  children: BookmarkItem[];  // 该分类下的书签
}

// 单个书签（二级）
interface BookmarkItem {
  id: string;           // chrome.bookmarks 的 bookmark id
  title: string;        // 网站名称
  url: string;          // 网址
  usageCount: number;   // 累计打开次数
  favicon?: string;     // 网站图标 URL（可选，用于星星纹理）
}

// 星星布局数据
interface StarLayout {
  bookmark: BookmarkItem;
  position: { x: number; y: number; z: number };
  size: number;         // 星星显示大小
  twinklePhase: number; // 闪烁相位（随机偏移）
}

// 路牌指示牌数据
interface SignboardData {
  category: BookmarkCategory;
  angle: number;        // 指示牌的旋转角度
  position: number;     // 在柱子上的高度位置
}
```

## 4. 后端设计

### 4.1 BookmarkService

```
职责：从 chrome.bookmarks API 读取并解析书签树
```

**核心方法：**

- `getBookmarkTree()` → 获取完整书签树
- `parseCategories(tree)` → 扁平化为 `BookmarkCategory[]`
  - 遍历书签栏，提取所有文件夹作为 Category
  - 每个文件夹下的直接书签子节点作为 BookmarkItem
  - 跳过嵌套超过一层的子文件夹（或将其扁平化处理）
- `getCategoryById(id)` → 按 id 获取单个分类

**实现要点：**
- `chrome.bookmarks.getTree()` 返回嵌套结构
- 只取"书签栏"节点（`parentId === "1"` 或 `rootId`）下的文件夹
- 忽略"其他书签"和"移动端书签"，可配置

### 4.2 UsageTracker

```
职责：追踪书签使用频率，用于后续星星大小调整
```

**核心方法：**

- `recordClick(bookmarkId)` → 记录一次点击
- `getUsageCount(bookmarkId)` → 查询使用次数
- `getAllUsageStats()` → 获取全部统计

**实现要点：**
- 使用 `chrome.storage.local` 持久化
- 数据结构：`{ [bookmarkId]: { count: number, lastClicked: timestamp } }`
- 前端每次打开书签时通知后端记录

### 4.3 MessageHandler

```
职责：Service Worker 中的消息路由
```

**支持的消息类型：**

| 消息类型 | 请求 | 响应 |
|---------|------|------|
| `GET_CATEGORIES` | 无参数 | `BookmarkCategory[]` |
| `GET_BOOKMARKS` | `{ categoryId }` | `BookmarkItem[]` |
| `RECORD_CLICK` | `{ bookmarkId }` | `void` |
| `GET_USAGE` | `{ bookmarkId }` | `{ count, lastClicked }` |

## 5. 前端设计

### 5.1 技术选型

| 层面 | 选择 | 理由 |
|------|------|------|
| 3D 渲染 | **Three.js** | 粒子系统（星星）、相机动画成熟；生态好 |
| 构建 | **Vite** + `@crxjs/vite-plugin` | 快速 HMR，天然支持 TS |
| 语言 | **TypeScript** | 类型安全，前后端共享类型 |
| 样式 | 无 CSS 框架 | 全 3D 渲染，仅 HTML canvas |

### 5.2 SceneManager

```
职责：管理场景生命周期、相机状态机、过渡动画编排
```

**相机状态机：**

```
                          ┌──────────────┐
                          │  CAMP_WIDE   │  初始广角，俯瞰营地全景
                          └──────┬───────┘
                                 │ 点击路牌
                                 ▼
                          ┌──────────────┐
                          │ SIGNPOST_FOCUS│  相机推进路牌，路牌放大为视觉焦点
                          └──────┬───────┘
                                 │ 点击某个指示牌（分类）
                                 ▼
                          ┌──────────────┐
                          │  CAT_WALKING │  小猫沿小路前行，相机跟随
                          └──────┬───────┘
                                 │ 小猫走到尽头
                                 ▼
                          ┌──────────────┐
                          │ CAT_TO_STAR  │  相机推向小猫 → 跟随小猫视线抬头
                          └──────┬───────┘
                                 │ 抬头完成
                                 ▼
                          ┌──────────────┐
                          │ STARRY_IDLE  │  星空交互状态
                          └──────┬───────┘
                                 │ 点击小路（返回）
                                 ▼
                          ┌──────────────┐
                          │  STAR_TO_CAMP│  反向过渡：低头 → 后退 → 回到营地
                          └──────────────┘
                                 │
                                 ▼
                          CAMP_WIDE（重新进入）
```

**核心方法：**
- `init()` → 初始化 Three.js（renderer, camera, 基础 lighting）
- `loadCampScene(categories)` → 构建场景一，进入 `CAMP_WIDE` 状态
- `focusSignpost()` → 相机推进路牌 → `SIGNPOST_FOCUS` 状态
- `startCatWalk(category)` → 小猫行走 + 相机跟随 → `CAT_WALKING` 状态
- `transitionToStarry()` → 相机推近猫 → 仰头 → `STARRY_IDLE` 状态
- `transitionBackToCamp()` → 反向动画 → `CAMP_WIDE` 状态
- `render()` → requestAnimationFrame 驱动的主渲染循环
- `update(deltaTime)` → 每帧根据当前状态更新相机、动画进度

### 5.3 CampScene（场景一：营地路牌）

**空间布局（俯视示意）：**

```
         Z轴 (远方)
         │
         │    小路延伸方向 →
         │    ═══════════════
         │         🐱
         │    (猫在小路中途)
         │
    ─────┼──────────────────── X轴 →
         │
         │        🪧
         │    路牌(中间偏右)
         │
         │   [相机初始位置]
         │   广角，俯瞰营地
```

**画面构成：**
- **背景**：渐变天空（暖色调，黄昏/傍晚感）
- **地面**：草地，覆盖整个底部
- **小路**：从画面中左向远方（Z 轴正方向）延伸，具有透视纵深感
- **路牌**：位于画面**中间偏右**（X > 0），木柱 + 多块指向不同方向的木牌
- **小猫（2D Sprite）**：站在小路中途，面向远方

**初始状态 — CAMP_WIDE：**
- 相机位置：营地后上方，广角俯瞰
- 可以看到完整的营地场景：草地、小路、路牌、小猫
- 能看清路牌上有多个指示牌，但文字不一定是阅读大小

**交互 — 点击路牌 → SIGNPOST_FOCUS：**
- Three.js Raycaster 检测到点击路牌
- 相机平滑推进（dolly in）向路牌
- 路牌在画面中变大，成为视觉焦点
- 指示牌上的分类文字现在清晰可读
- 小猫仍在画面中（可能在路牌旁边或稍远处）

**交互 — 点击指示牌 → CAT_WALKING：**
- Three.js Raycaster 检测到点击某块指示牌
- 被点击的指示牌高亮（短暂发光或抖动）
- 触发过渡动画

**技术方案：纯 3D（Three.js）**
- 所有场景元素用几何体 + 材质构建
- 低多边形（Low Poly）风格降低建模难度

**Signpost 实现细节：**
- 柱子：`CylinderGeometry(0.08, 0.1, 2.5)` + 木纹材质（`MeshStandardMaterial` + 棕色）
- 指示牌：`BoxGeometry(0.6, 0.15, 0.03)` 多块，绕柱旋转不同角度
- 每块指示牌的朝向随机在 ±45° 范围内分布
- 文字通过 Canvas 2D 动态绘制到纹理，贴到指示牌正面
- 每块指示牌作为独立的 `Mesh`，可被 Raycaster 检测点击

### 5.4 StarryScene（场景二：星空书签）

```
画面元素空间位置（XYZ 世界坐标）：

   Y (上)
   │  ⭐ ⭐ ⭐  ⭐   ⭐    ← 星星层 (y: 8~25, xz 散布在上方天空区域)
   │ ⭐  ⭐  ⭐    ⭐ ⭐
   │   ⭐    ⭐  ⭐  ⭐
   │
   │  🌙 可选月亮 (y: 18, 远处)
   │
───┼────────────────────── X (右)
   │  🐱 小猫坐姿 (底部中央, z≈18, y≈0.5)
   │  ═══ 小路 (z: 0→20, 猫身后延伸)
   │  ░░░ 草地 (y≈0, 大面积)
```

**画面构成（屏幕视角）：**
- 上方 70%：深蓝黑渐变天空，布满星星（粒子系统）
- 下方 30%：暗色草地轮廓，小猫剪影坐在中央偏下
- 猫身后隐约可见小路（返回的视觉线索）

**StarField 实现细节：**

- 使用 Three.js `Points` + `BufferGeometry` 构建粒子系统
- 星空粒子数量 = 当前分类的书签数
- 星星散布于上半球区域（y: 8~25, xz 在相机前方半球）
- 每颗星的位置预先随机生成并存储，避免每帧抖动
- 星星大小随机范围：0.05 ~ 0.25（世界单位），对应屏幕像素约 3~15px
- 闪烁通过逐帧微调 opacity（使用 `PointsMaterial.vertexColors` 或更新 `size` attribute）

**备选 — 星形 Sprite：**
如果 `Points` 的圆形粒子不够"像星星"，改用多个 `THREE.Sprite`，每个贴一颗四角星/五角星纹理（Canvas 绘制的小星形图）。代价是 draw call 增多，但书签通常在 50 个以内，性能可接受。

**星星点击检测：**
- Three.js Raycaster 对 `Points` 的支持有限（只能检测点位置）
- **推荐方案**：使用 `Sprite` 代替 `Points`，Raycaster 对 Sprite 友好
- 或者手动计算：鼠标屏幕坐标 → 反投影到天空半球 → 找最近的星星（距离阈值判断点击）

**Interaction（点击星星）：**
- 点击 → 星星短暂放大 1.5x 并高亮（~200ms）→ 通过 `window.location.href = url` 在当前标签页跳转，或 `window.open(url)` 新标签页
- 同时通知 Service Worker 记录本次点击（频率统计）

**Hover 效果：**
- 鼠标悬停星星 → 星星 scale 变为 1.3x + 外围发光圈 + CSS overlay tooltip 显示网站名
- 移开 → 恢复原大小

**Interaction（返回营地）：**
- 点击小猫身后的小路区域（或直接点击猫身后的地面隐式区域）
- → `SceneManager.transitionBackToCamp()` 触发反向过渡

### 5.5 过渡动画：完整的镜头叙事

**整体思路：单场景连续过渡。** 所有元素（营地、路牌、星星）在同一个 Three.js 场景中，通过相机运动 + 光照变化完成场景切换。星星从一开始就存在于场景上方高处，在场景一中因为相机朝向地面 + 白天光照而不可见。

---

#### 阶段 0：CAMP_WIDE → SIGNPOST_FOCUS（~1 秒）

用户点击路牌 → 相机推进路牌：

| 维度 | 变化 |
|------|------|
| 相机 position | 从广角位置沿 Z 轴推向路牌（路牌在 X 轴偏右） |
| 相机 lookAt | 锁定路牌中心 |
| FOV | 可微调缩小（制造"聚焦"感） |
| 动画曲线 | easeInOutCubic |

相机到达新位置后，路牌占据画面显著比例，指示牌文字清晰可读。

---

#### 阶段 1：CAT_WALKING — 小猫前行（~2.5 秒）

用户点击某块指示牌 → 小猫沿小路前进，相机旋转跟随：

```
起始（路牌焦点）:
  camera.position: 路牌前方，略俯视
  camera.lookAt:   路牌

结束（跟到小路尽头）:
  camera.position: 猫身后上方
  camera.lookAt:   前方小路的消失点
```

| 时间 | 画面 |
|------|------|
| 0.0s | 路牌上的目标指示牌高亮 → 小猫从当前位置起身 |
| 0.3s | 小猫走到小路中央，开始沿 Z 轴正方向前进 |
| 0.3-2.2s | 小猫持续前行（走步循环动画），相机从路牌焦点**旋转并平移**到猫身后 |
| 2.2s | 小猫到达小路尽头，停下 |
| 2.5s | 小猫转为"坐下"姿态 |

**相机运动路径：** 从路牌前方弧线过渡到猫身后上方——相机在 XZ 平面上画一条弧线，同时 Y 轴缓慢下降（更贴近猫的视角高度）。

**动画曲线：** easeInOutCubic，起和止都柔和。

---

#### 阶段 2：CAT_TO_STAR — 抬头看星空（~2 秒）⭐ 最关键的过渡

小猫坐下后，镜头叙事分两步：

**Step A — 推向小猫（~0.8s）：**
- 相机从猫身后方向**向前推近**猫
- 猫在画面中变大，成为画面焦点
- 略微降低相机高度，接近猫的视线水平

**Step B — 跟随猫的视角抬头（~1.2s）：**
- 相机 lookAt 从水平前方逐渐**旋转向上**（模拟猫抬头、人也跟着抬头）
- 相机 lookAt 的 Y 值从 `0.5`（略低头看路）上升到 `20`（仰望星空）
- 同时：

| 维度 | 变化 |
|------|------|
| **环境光** | 暖色 `#FFE4B5` → 冷暗色 `#0A0A2E` |
| **方向光（太阳）** | 亮度 `1.0` → `0` |
| **天空背景** | 暖黄橙渐变 → 深蓝黑星空渐变 |
| **星星** | opacity/size `0` → `1`，伴随微弱闪烁 |
| **地面/草** | 逐渐变暗融入夜色 |
| **猫** | 切换到"仰头坐姿" Sprite 帧 |

**为什么星星"自然出现"：**
- 星星从一开始就在场景上方（y: 10~30）
- 场景一相机向下看（lookAt.y ≈ 0~0.5），星星在视锥体上方外
- camera lookAt 仰角增大 → 星星进入视锥体 → 自然出现在画面中
- 配合光照变暗 → 星星在暗背景中凸显

---

#### 阶段 3：STARRY_IDLE — 稳定（持续）

- 相机微微上下浮动（呼吸感，幅度 0.05 单位）
- 星星保持微弱闪烁
- 猫保持仰头坐姿，尾巴偶尔摆动
- 用户可以 hover/点击星星

---

#### 反向过渡：STAR_TO_CAMP（~3 秒）

用户点击小猫身后的小路 → 反向播放：

1. 相机 lookAt 从天空下降回水平
2. 光照从暗变暖
3. 星星淡出
4. 小猫起身
5. 相机从猫身后弧线后退回到路牌焦点
6. 相机再从路牌后退回到 CAMP_WIDE 广角

动画曲线全程使用 easeInOutCubic，让往返过渡都柔和自然。

---

#### 动画实现方案

使用自定义 Tween 系统（避免引入 GSAP 作为依赖）：

```typescript
// 简单的 easing + 时间插值工具
class Tween {
  constructor(
    target: THREE.Vector3 | THREE.Euler,
    from: any,
    to: any,
    duration: number,
    easing: (t: number) => number,
    onUpdate?: () => void,
    onComplete?: () => void
  ) {}
  update(dt: number): boolean { /* 返回 true 表示完成 */ }
}
```

每一帧的 `SceneManager.update(dt)` 驱动所有活跃的 Tween 实例。也可以直接用 Three.js 内置的 `THREE.AnimationMixer` 配合关键帧轨道。

### 5.6 小猫角色（2D Sprite）

```
设计约束：
  - 2D 表现，非 3D 模型
  - 橘猫配色
  - 尾巴需灵动、有表现力（核心特征）
```

**实现方案：**

小猫主体使用 **Canvas 2D 绘制的 Sprite 序列帧**：

- 用离屏 Canvas 逐帧绘制猫的不同姿态
- 导出为 `THREE.CanvasTexture`，贴到 `THREE.Sprite` 上
- 根据动画状态切换 Sprite 的纹理

**猫的绘制层（Canvas 合成）：**

```
┌──────────────────┐
│   耳朵 (三角形)    │
│  ┌──────────────┐ │
│  │   头部 (圆)   │ │
│  │  ⚭ ⚭ 眼睛   │ │
│  │   ▲ 鼻子     │ │
│  └──────────────┘ │
│  身体 (椭圆)       │
│  四肢 (线条)       │
│                   │
│  ~~~~~ 尾巴 ~~~~~ │  ← 独立绘制的动画层
└──────────────────┘
```

**尾巴的独立动画：**

为了"灵动"的尾巴，**尾巴作为独立的 Sprite** 叠加在身体 Sprite 上。这允许尾巴有不依赖于身体的动画：

- **悠闲摆动**（待机）：正弦波驱动的左右摆动
- **跟随甩动**（行走）：与步伐节奏同步的反向摆动
- **轻快弹动**（坐下）：随机的小幅 flick，表现好奇心
- **缓慢画圈**（仰望星空时）：尾巴尖慢慢画圈，沉浸放松

尾巴 Sprite 围绕身体后部的锚点旋转，旋转角度由 sine + 随机扰动混合驱动。

**动画状态与帧：**

| 状态 | Sprite | 尾巴动画 |
|------|--------|---------|
| 待机（idle） | 侧面站立 | 悠闲摆动 |
| 行走（walk） | 4 帧走步循环 | 跟随甩动 |
| 坐下（sit） | 坐姿侧面 | 轻快弹动 |
| 仰头坐（sit_look_up） | 坐姿 + 头抬起 45° | 缓慢画圈 |

**Sprite 始终面对相机（Billboarding）：**
- `THREE.Sprite` 天然 billboard，始终正对相机
- 不需要额外的朝向计算
- 侧面视角的猫 Sprite + billboard 会有"纸片感"，但在俯视/后跟随的相机角度下不违和

**备选方案 — 如果需要多角度猫：**
- 如果后续发现 billboard 的纸片感太强，可以绘制 3 个角度的 Sprite（正面/侧面/背面），根据相机与猫的相对角度切换

### 5.7 渲染管线（每个 frame）

```
1. 更新 Cat 动画（walk cycle 或 idle）
2. 如果在过渡中 → 更新 TransitionController
3. 更新 StarField 闪烁（如果在星空场景）
4. renderer.render(scene, camera)
5. requestAnimationFrame(loop)
```

## 9. 关键风险与缓解

| 风险 | 缓解策略 |
|------|---------|
| `chrome.bookmarks` API 返回空或权限问题 | 提供空状态提示 + demo 数据降级 |
| Three.js 在扩展环境加载慢 | CDN 外部加载 或 Tree-shaking 减小包体积 |
| 低多边形模型不美观 | 找开源 low-poly 模型参考，或用简单几何体组合 |
| 新标签页加载性能 | 延迟加载非关键资源，首屏优先渲染 |
| 书签数量多时星星密集 | 设置显示上限（如 50 颗），其余随机抽样或分页 |
